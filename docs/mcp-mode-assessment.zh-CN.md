# MCP 模式改造评估与可实施性报告

## 1. 报告目的

本报告用于评估将当前仓库所定义的 Agents Knowledge DB 治理内核演进为统一 `MCP Server` 模式的可行性，并重点分析其对以下三类角色的影响：

- 数据库系统维护角色
- 数据仓库维护角色
- 数据仓库使用角色

本报告同时给出可实施的落地路径、阶段性目标、风险控制要求与推荐决策。

## 2. 评估对象与边界

### 2.1 当前仓库基线

依据当前仓库设计，系统已经具备 MCP 化改造的关键基础：

- 单一规则源：`RULES.md`
- 兼容适配入口：`AGENTS.md`、`CLAUDE.md`、`GEMINI.md`
- 机器事实层：`.knowledge-registry/`
- 派生扫描层：`DBMS/index/`
- 行为契约层：`skills/`
- 双仓模式：system repo 发布治理内核，data repo 消费只读快照

这意味着当前系统已经天然具备从“文档驱动治理”演进到“协议化治理服务”的条件。

### 2.2 MCP 模式的目标定义

本报告假设的 “MCP 模式” 指：

- 以一个统一的 `MCP Server` 暴露治理能力
- 将规则、注册表、索引、审计、快照升级、主题维护等能力通过 `tools`、`resources`、`prompts` 对外提供
- 由不同客户端或代理以角色受控方式接入
- 保持 “注册表是真源，索引是派生态” 这一核心治理边界不变

### 2.3 不在本次评估范围内

- 具体 SDK 选型
- 具体宿主产品 UI 设计
- 多租户计费与商业化方案
- 非知识库场景下的通用数据库访问代理

## 3. 结论摘要

### 3.1 总体结论

结论为：**可以实施，但必须采取分阶段、分角色、分权限的受控改造策略。**

如果直接将当前仓库能力粗暴封装成一个“任何代理都可写”的 MCP 服务，会带来以下高风险问题：

- 规则解释权被工具调用路径绕开
- 注册表与索引被误当作同级真源
- 数据仓维护职责与数据库系统维护职责混淆
- 多代理并发写入导致状态不一致
- 审计链条断裂，变更责任难以追溯

因此，推荐采用如下原则：

- `resources` 暴露规则、状态、索引、报告
- `tools` 只暴露经过权限分级的操作
- 高风险写操作默认只允许数据库系统维护角色或其代理执行
- 数据仓库维护角色以“受限写 + 审批流”方式接入
- 数据仓库使用角色默认采用只读、检索、解释、提问式交互

### 3.2 可实施性评级

| 维度 | 评级 | 说明 |
|---|---|---|
| 技术可实施性 | 高 | 当前仓库已有规则源、机器契约、脚本、索引与角色模型 |
| 架构适配性 | 高 | 现有结构天然适合映射为 MCP `tools/resources/prompts` |
| 组织可实施性 | 中 | 角色边界需要重新定义并固化到权限模型中 |
| 运维可实施性 | 中 | 需要增加认证、审计、并发控制、幂等设计 |
| 数据安全可控性 | 中 | 可控，但前提是高风险写操作强隔离 |
| 近期落地建议 | 高 | 建议先做只读 MVP，再逐步开放受控写入 |

## 4. 当前治理模型与 MCP 的映射关系

### 4.1 从治理资产到 MCP 原语的映射

| 当前资产 | 当前作用 | 推荐 MCP 映射 |
|---|---|---|
| `RULES.md` | 人类可读规则源 | `resource` |
| `AGENTS.md` / `CLAUDE.md` / `GEMINI.md` | 平台适配入口 | `resource` |
| `.knowledge-registry/*` | 机器事实真源 | 只读 `resource` + 受控写 `tool` |
| `DBMS/index/*` | 派生扫描视图 | `resource` |
| `scripts/rebuild_dbms_index.py` | 索引重建工具 | 运维型 `tool` |
| `scripts/validate_data_repo.py` | 仓库校验工具 | 审计型 `tool` |
| `skills/*` | 行为契约 | `prompt` 或外部工作流说明 `resource` |
| `templates/*` | 采纳与安装资产 | `resource` + 安装型 `tool` |

### 4.2 推荐 MCP 服务边界

统一 MCP 服务不应等价于“统一写入口”，而应等价于“统一协议入口 + 分层能力域”。

推荐拆分为以下能力域：

- `governance-read`
  - 读取规则、注册表、索引、状态、报告
- `registry-admin`
  - 注册主题、修复路径、更新注册表条目
- `dbms-admin`
  - 重建索引、执行审计、输出发现
- `workflow-ops`
  - intake、curation、promotion、archive 等流程任务
- `system-snapshot-admin`
  - 比较系统快照、提出升级建议、执行应用

推荐在实现上使用一个 MCP Server 进程，但在逻辑上保持以上域隔离，并对不同客户端返回不同的工具清单。

## 5. 对三类角色的影响评估

## 5.1 数据库系统维护角色

本角色对应当前仓库中的系统维护者、治理内核维护者，以及 `db-admin-agent` 背后的责任主体。它关注的是控制平面、状态平面、索引派生层与系统快照升级安全，而不是日常知识增长。

### 5.1.1 角色变化

| 维度 | MCP 化前 | MCP 化后 |
|---|---|---|
| 职责定位 | 维护规则、接口、模式、脚本、快照机制 | 额外负责 MCP 服务能力边界、工具契约、认证授权与审计链 |
| 主要操作对象 | system repo、模板、schemas、验证脚本 | 再加 MCP tool schema、resource 暴露策略、客户端兼容性 |
| 风险承担 | 规则漂移、模板失配、升级错误 | 再加误授权、越权写入、工具语义失真、跨角色污染 |
| 运维复杂度 | 中 | 高 |

### 5.1.2 正向影响

- 规则与机器能力可以统一发布，降低不同代理生态下的接入差异。
- 工具输入输出可以进一步结构化，减少仅靠自然语言阅读规则带来的歧义。
- 可以把当前脚本能力正式产品化，形成稳定的治理 API 面。
- 能更容易做自动化回归测试、权限回归测试、升级前后兼容测试。

### 5.1.3 负向影响

- 系统维护者将从“维护治理内核”升级为“维护治理内核 + 协议服务 + 权限系统”。
- 一旦工具定义不严谨，客户端可能绕过原来的阅读顺序，直接发起危险操作。
- 如果不做角色隔离，`db-admin-agent` 的特权能力会被普通使用角色滥用。

### 5.1.4 对该角色的关键要求

- 必须定义角色到工具的授权矩阵
- 必须为高风险写操作加入审批或确认机制
- 必须维护工具的输入输出 schema 与版本兼容策略
- 必须确保索引层永远不能覆盖注册表真源
- 必须记录每次敏感操作的调用人、参数摘要、结果与 ledger 事件

### 5.1.5 综合评估

| 评估项 | 结果 |
|---|---|
| 角色收益 | 高 |
| 负担增加 | 高 |
| 适合优先纳入 MCP | 是 |
| 前提条件 | 需要最先完成权限、审计、幂等与变更追踪设计 |

## 5.2 数据仓库维护角色

本角色对应数据仓库或数据仓库运营者，负责 data repo 内的主题维护、日常治理执行、局部修复、索引消费与工作流推进。该角色更接近“业务侧运维者”，而不是系统内核维护者。

### 5.2.1 角色变化

| 维度 | MCP 化前 | MCP 化后 |
|---|---|---|
| 进入方式 | 读规则文件、读注册表、按技能执行 | 通过 MCP 客户端调用受限工具 |
| 工作方式 | 文件导向、规则导向 | 协议导向、任务导向 |
| 能力边界 | 较依赖文档理解深度 | 更依赖服务显式暴露的工具边界 |
| 运维效率 | 中 | 中到高 |

### 5.2.2 正向影响

- 日常维护动作更标准化，例如索引审计、主题注册补全、路径异常修复等。
- 维护人员不需要每次都从底层文件结构推断可执行步骤，降低上手成本。
- 可以通过 `resources` 快速读取当前仓库状态、主题摘要、扫描发现，提高问题定位速度。
- 更适合与多代理协同，因为调用入口会比纯文档式协作更一致。

### 5.2.3 负向影响

- 如果工具能力设计过宽，维护角色可能误触系统级操作。
- 如果工具能力设计过窄，维护角色又会回退到直接改文件，形成双轨治理。
- 数据仓维护者会更依赖 MCP 服务可用性；一旦服务不可用，必须保留文档与文件级后备流程。

### 5.2.4 对该角色的关键要求

- 允许读取全部治理必需 `resources`
- 允许执行低风险或中低风险维护工具
- 对注册表修改应采用受限写策略
- 对系统快照升级、批量修复、跨层迁移等操作默认无权直写
- 对每次维护任务自动生成 report 与 ledger 记录

### 5.2.5 综合评估

| 评估项 | 结果 |
|---|---|
| 角色收益 | 高 |
| 负担增加 | 中 |
| 适合优先纳入 MCP | 是 |
| 前提条件 | 需要明确“数据维护”和“系统维护”的工具边界 |

## 5.3 数据仓库使用角色

本角色指知识库最终使用者、主题查询者、分析使用者、只读型业务代理或决策支持代理。其核心诉求是“可靠获取上下文、提出问题、获得解释”，而不是维护治理结构。

### 5.3.1 角色变化

| 维度 | MCP 化前 | MCP 化后 |
|---|---|---|
| 主要能力 | 读文件、检索主题、人工理解上下文 | 通过 MCP 直接获取结构化上下文和限定工具 |
| 学习成本 | 相对较高 | 显著降低 |
| 误操作风险 | 低到中 | 若默认只读则低 |
| 价值释放速度 | 中 | 高 |

### 5.3.2 正向影响

- 使用者可以通过统一入口检索规则、主题摘要、注册状态、来源线索。
- LLM 或代理更容易获得结构化上下文，而不是在大量文件中自行推断。
- 更适合做问答、解释、报告生成、覆盖率查看、问题发现等场景。
- 可以显著提升“用库”的体验，而不破坏“管库”的秩序。

### 5.3.3 负向影响

- 如果对使用者暴露写工具，会把查询型角色变成潜在破坏源。
- 如果结果不明确区分“真源事实”和“派生发现”，使用者会误读索引结果。
- 如果没有资源分级与敏感字段控制，可能暴露不应直接外显的内部状态。

### 5.3.4 对该角色的关键要求

- 默认只读
- 默认无注册表写权限
- 默认无索引重建权限
- 输出中必须明确标注数据来源：规则、注册表、索引、报告、派生结论
- 支持资源级过滤，避免将系统内部运维状态直接暴露给普通使用者

### 5.3.5 综合评估

| 评估项 | 结果 |
|---|---|
| 角色收益 | 很高 |
| 负担增加 | 低 |
| 适合优先纳入 MCP | 是，且应最先作为 MVP 用户群 |
| 前提条件 | 只读隔离必须先做完 |

## 6. 架构可实施性分析

## 6.1 为什么当前仓库适合改造成 MCP

当前仓库已经具备典型的协议化前提：

- 有明确规则源，适合转为 `resource`
- 有 machine-readable schemas，适合转为工具输入输出约束
- 有脚本与验证器，适合封装为执行型 `tool`
- 有角色与任务类型，适合转为授权模型与提示模板
- 有双仓和快照边界，适合划分控制平面与数据平面

这意味着 MCP 改造不是“从零做一层协议”，而是“把已存在的治理契约结构化并服务化”。

## 6.2 推荐的 MCP 能力设计

### 6.2.1 推荐 `resources`

- `governance://rules/root`
- `governance://rules/system-snapshot`
- `governance://registry/vault`
- `governance://registry/agent-roster`
- `governance://registry/promotion-queue`
- `governance://registry/change-ledger/latest`
- `governance://dbms/index/file-summary`
- `governance://dbms/index/findings`
- `governance://reports/latest-audit`
- `governance://schemas/*`

### 6.2.2 推荐 `tools`

- `get_topic_context`
- `list_topic_findings`
- `validate_data_repo`
- `rebuild_dbms_index`
- `propose_registry_update`
- `apply_registry_update`
- `create_promotion_proposal`
- `review_snapshot_upgrade`
- `apply_snapshot_upgrade`
- `generate_governance_report`

### 6.2.3 推荐 `prompts`

- `onboard_agent_to_vault`
- `review_topic_health`
- `prepare_promotion_review`
- `run_monthly_audit`
- `analyze_registry_drift`

## 6.3 最重要的架构约束

### 约束一：注册表必须保持真源地位

无论 MCP 如何封装，`.knowledge-registry/` 仍然是：

- topic identity 真源
- layer placement 真源
- role authority 真源
- promotion state 真源
- change history 真源

任何 `tool` 都只能通过受控流程修改它，绝不能让 `DBMS/index/` 或客户端缓存反向覆盖它。

### 约束二：索引层必须保持派生地位

`DBMS/index/` 只能是：

- 审计视图
- 扫描视图
- 发现视图
- 性能优化视图

它不能被描述成“第二事实库”。

### 约束三：高风险操作必须人机共管

以下操作不应默认自动执行：

- 批量注册表修改
- 系统快照应用
- 大范围路径迁移
- canonical 区域修复
- protected zone 清理

这些操作应要求显式确认、审批人记录或受控流水线。

### 约束四：必须保留文件级后备路径

MCP 服务是治理增强层，不是唯一生存层。仓库仍应能在没有 MCP 服务时依靠：

- `RULES.md`
- 注册表文件
- 校验脚本
- 索引重建脚本

完成基本治理恢复。

## 7. 风险评估

| 风险 | 影响 | 概率 | 风险等级 | 缓解措施 |
|---|---|---|---|---|
| 工具越权导致错误写入 | 高 | 中 | 高 | 角色授权矩阵、只读默认、敏感工具二次确认 |
| 索引被误当真源 | 高 | 中 | 高 | 所有输出显式标注 source-of-truth 与 derived-state |
| 多代理并发写冲突 | 高 | 中 | 高 | 乐观锁、版本号、写前校验、幂等操作设计 |
| 文档规则与工具行为不一致 | 高 | 中 | 高 | schema 驱动开发、回归测试、变更联动审查 |
| MCP 服务不可用造成流程中断 | 中 | 中 | 中 | 保留脚本与文件后备路径 |
| 快照升级绕过 DBA 评审 | 高 | 低到中 | 中高 | 将 snapshot apply 设为特权工具 |
| 审计链断裂 | 高 | 中 | 高 | 强制 ledger 记录与 report 输出 |
| 普通用户接触敏感资源 | 中到高 | 中 | 中高 | 资源分级、字段脱敏、按角色列资源 |

## 8. 分阶段实施路线

## 8.1 第一阶段：只读 MVP

目标：先把“好用”和“安全”建立起来，不先追求可写。

范围：

- 暴露规则、注册表、索引、schema 为 `resources`
- 提供查询型工具
- 提供主题上下文聚合工具
- 不开放任何直接写工具

适用角色：

- 数据仓库使用角色
- 数据仓库维护角色中的只读分析场景

验收标准：

- 能稳定返回 topic 上下文
- 能区分真源与派生态
- 能输出结构化报告

## 8.2 第二阶段：受限维护工具

目标：开放低风险维护能力，但保持系统平面隔离。

范围：

- `validate_data_repo`
- `list_topic_findings`
- `propose_registry_update`
- 只允许局部、可审计、可回滚的注册表修改建议

适用角色：

- 数据仓库维护角色

验收标准：

- 所有写入都有 ledger
- 所有提议都能追踪发起者与依据
- 不能越权执行系统级操作

## 8.3 第三阶段：DBMS 运维能力

目标：让数据库系统维护角色通过 MCP 管理索引与审计。

范围：

- `rebuild_dbms_index`
- `generate_governance_report`
- `analyze_registry_drift`
- `run_monthly_audit`

适用角色：

- 数据库系统维护角色

验收标准：

- 索引重建结果一致
- 报告结构稳定
- 失败可回溯

## 8.4 第四阶段：高风险受控写入

目标：逐步开放快照升级、批量修复、promotion 流程。

范围：

- `apply_registry_update`
- `review_snapshot_upgrade`
- `apply_snapshot_upgrade`
- `create_promotion_proposal`

适用角色：

- 数据库系统维护角色
- 经授权的数据仓库维护角色

验收标准：

- 全量审计可追溯
- 有审批或确认
- 有回滚与重试策略

## 9. 推荐的角色授权矩阵

| 工具/能力 | 数据库系统维护角色 | 数据仓库维护角色 | 数据仓库使用角色 |
|---|---|---|---|
| 读取规则与 schema | 允许 | 允许 | 允许 |
| 读取注册表 | 允许 | 允许 | 允许，必要时脱敏 |
| 读取索引与 findings | 允许 | 允许 | 允许，建议限制细节级别 |
| 主题上下文查询 | 允许 | 允许 | 允许 |
| 数据仓校验 | 允许 | 允许 | 不允许 |
| 提议注册表更新 | 允许 | 允许 | 不允许 |
| 应用注册表更新 | 允许 | 条件允许 | 不允许 |
| 索引重建 | 允许 | 条件允许 | 不允许 |

## 10. 实施建议

### 10.1 推荐决策

建议立项，但采用以下策略：

- 先做统一 MCP 协议入口
- 不先做统一写入口
- 第一版先服务“数据仓库使用角色”和“数据仓库维护角色的只读/低风险场景”
- `db-admin-agent` 专属能力后置，并保持强隔离

### 10.2 推荐的最小可行版本

MVP 建议只包含：

- 规则与 schema `resources`
- 注册表与索引 `resources`
- `get_topic_context`
- `list_topic_findings`
- `validate_data_repo`
- `generate_governance_report`

这样可以先验证三件事：

- 客户端是否真能更稳定理解仓库治理结构
- 使用角色是否明显降低学习成本
- 维护角色是否能提高问题定位效率

### 10.3 不推荐的做法

- 不推荐一开始就开放注册表直接写入
- 不推荐让普通使用角色看见运维型工具
- 不推荐让索引输出承担真源语义
- 不推荐跳过 ledger 与 report 直接修改仓库状态
- 不推荐让 system repo 与 data repo 通过 MCP 变成无边界直连

## 11. 最终判断

从当前仓库的治理成熟度来看，MCP 化不是概念性尝试，而是有明确工程基础的下一步演进。

它最适合带来的提升不是“让所有人都能改”，而是：

- 让使用角色更容易用
- 让维护角色更容易管
- 让系统维护角色更容易审

因此，本项目的最佳推进方式不是“全面服务化”，而是“治理优先的协议化”：

- 保持规则源稳定
- 保持注册表真源地位
- 保持索引派生地位
- 保持角色授权清晰
- 保持高风险动作可审计、可追溯、可回退

在这些前提下，统一 MCP Server 方案具备较高可实施性，并值得进入设计与原型阶段。
